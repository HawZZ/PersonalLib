车载边缘计算不是单纯的“把算力放到车上”。从产品经理视角看，它是一套围绕 实时性、安全性、数据闭环、成本效率和体验稳定性 的产品系统。

这篇笔记的目标不是替代算法工程师或系统架构师，而是帮助产品经理理解：车载边缘计算为什么存在、服务哪些场景、如何拆需求、如何定义指标、如何设计产品路线图，以及在跨团队协作时该问哪些关键问题。

车端 传感器 / ECU / 网关 / 域控制器 实时感知、本地规则、缓存、控制
路侧 RSU / 边缘服务器 / 信号控制器 协同感知、区域决策、低延迟分发
云端 数据平台 / OTA / 训练 / 运营 全局分析、模型更新、车队管理

1. 产品经理为什么要学习车载边缘计算

产品经理学习车载边缘计算，核心不是掌握所有底层算法，而是建立产品判断能力。

1.1 需要回答的问题

  • 这个能力解决的是安全问题、效率问题、体验问题，还是成本问题？
  • 哪些计算必须在车端完成，哪些可以放到云端？
  • 数据延迟、带宽、功耗、稳定性、隐私之间如何取舍？
  • 车端边缘计算能力如何产品化，而不是停留在技术方案？
  • 车端、路侧、云端、运维平台之间的边界在哪里？
  • 需求失败时，用户看到的是功能不可用、体验下降，还是安全风险？

1.2 产品经理需要形成的判断

判断维度关键问题产品含义
实时性是否必须毫秒级响应？决定计算放在车端还是云端
安全性失败后是否影响驾驶安全？决定功能等级、冗余和降级策略
可靠性弱网、断网、传感器异常时怎么办？决定边缘自治能力
成本算力、传感器、通信、云资源成本是否可控？决定商业可行性
数据闭环车端数据如何回流优化算法和产品？决定持续迭代能力
合规隐私是否涉及位置、驾驶行为、车内数据？决定数据采集和使用边界

2. 基础概念

2.1 什么是车载边缘计算

车载边缘计算是把数据处理、决策、控制或预处理能力放在车辆本地或靠近车辆的边缘节点上，而不是完全依赖远端云服务器。

常见计算位置：

  • 车端：域控制器、中央计算平台、智能座舱主机、T-Box、网关、传感器模组。
  • 路侧：RSU、边缘服务器、交通信号控制器、园区边缘节点。
  • 云端：训练平台、车队管理平台、OTA 平台、数据湖、仿真平台。

车载边缘计算的本质是：

把对延迟、可靠性、隐私或带宽敏感的任务放到更靠近数据源的位置执行。

2.2 与云计算、端侧计算的区别

类型位置优势局限
云计算数据中心算力强、资源弹性好、适合训练和全局分析延迟高、依赖网络、传输成本高
端侧计算传感器或车内设备响应快、隐私好、弱网可用算力和功耗受限
边缘计算车端或路侧边缘节点平衡实时性和协同能力架构复杂、运维难度高

相关基础概念可参考：边缘计算基本概念 和 车载边缘计算的特点。

3. 典型业务场景

智能驾驶毫秒级响应、误报漏报、接管逻辑。
V2X车路协同、弱网策略、可信身份。
车队运营成本、效率、告警闭环、远程诊断。
车载网关CAN / LIN / Ethernet 数据采集与协议转换。
智能座舱本地推理、隐私、OTA、多人权限。

3.1 智能驾驶与辅助驾驶

典型场景：

  • 障碍物识别
  • 车道线检测
  • 行人和车辆识别
  • 自动泊车
  • 驾驶员监测
  • 碰撞预警
  • 自动紧急制动

产品关注点：

  • 是否涉及安全闭环？
  • 延迟目标是多少？
  • 误报和漏报哪个风险更高？
  • 功能失败后如何提醒用户？
  • 是否需要降级到人工接管？
  • 是否需要记录触发证据用于复盘？

3.2 车联网与 V2X

典型场景：

  • 车辆与车辆通信，V2V。
  • 车辆与路侧基础设施通信，V2I。
  • 车辆与云平台通信，V2C。
  • 车辆与行人或移动设备通信，V2P。

产品关注点：

  • 这个场景是提升安全、提升通行效率，还是提升运营管理能力？
  • 数据来自车端、路侧，还是云端？
  • 弱网或无网时是否允许功能继续工作？
  • 多来源信息冲突时以谁为准？
  • 是否需要做可信身份认证？

3.3 商用车与车队运营

典型场景：

  • 车辆状态监控
  • 油耗和能耗分析
  • 驾驶行为分析
  • 预测性维护
  • 远程诊断
  • 车辆调度
  • 货运冷链监控

产品关注点：

  • 运营方最关心成本、效率、合规还是安全？
  • 哪些数据需要实时上报，哪些可以批量上传？
  • 车辆离线后是否需要本地缓存？
  • 告警如何分级？
  • 运维人员如何闭环处理问题？

3.4 车载网关与数据采集

典型场景：

  • CAN / LIN / Ethernet 数据采集
  • 传感器数据预处理
  • 协议转换
  • 本地规则判断
  • 数据压缩和过滤
  • 云端遥测上报

产品关注点：

  • 采集哪些信号？频率是多少？
  • 数据是否要原始上传，还是边缘侧先聚合？
  • 是否需要支持多车型、多协议、多配置？
  • 数据字典和信号版本如何管理？
  • 采集异常如何定位？

3.5 智能座舱

典型场景：

  • 语音识别
  • 车内摄像头感知
  • 个性化推荐
  • 本地多模态交互
  • 疲劳驾驶提醒
  • 座舱场景联动

产品关注点：

  • 哪些数据不能上传云端？
  • 本地推理延迟是否影响交互体验？
  • 模型升级是否通过 OTA？
  • 用户是否能关闭个性化数据采集？
  • 车内多人场景如何处理身份和权限？

4. 用户和角色

车载边缘计算产品通常不是单一用户产品，而是多角色系统。

角色关注点产品需求
驾驶员安全、易用、稳定、少打扰清晰提示、低误报、可理解的接管逻辑
车主车辆状态、成本、隐私状态可视化、远程诊断、数据授权
车队运营方效率、成本、风险车辆监控、告警闭环、报表分析
运维人员故障定位、远程处理日志、诊断、OTA、配置管理
主机厂平台化、可扩展、质量车型适配、能力复用、数据闭环
算法团队数据、算力、评估指标数据采集、模型部署、在线评估
硬件团队BOM、功耗、散热、接口算力选型、传感器接口、冗余设计

产品经理要做的是把这些角色的目标翻译成可落地的产品能力，而不是只描述一个技术模块。

5. 产品架构理解

5.1 车端到云端的典型架构

传感器 / ECU
  -> 车载网关 / 域控制器 / 中央计算平台
  -> 本地感知、融合、规则、缓存、控制
  -> T-Box / 5G / Wi-Fi / Ethernet
  -> 云平台 / 数据平台 / 运营平台
  -> OTA / 模型更新 / 策略配置 / 数据分析

产品经理需要关注三个边界：

  • 数据边界：哪些数据在车端处理，哪些上传云端。
  • 控制边界：哪些指令可以远程下发，哪些必须本地闭环。
  • 责任边界：功能失败时，车端、云端、网络、传感器分别承担什么责任。

5.2 车端模块

模块作用产品关注点
传感器接入摄像头、雷达、IMU、GPS、CAN 等数据输入接口、频率、质量、同步
数据预处理清洗、滤波、时间同步、格式转换数据质量和延迟
感知与融合目标识别、多传感器融合准确率、召回率、误报率
本地规则阈值判断、告警策略、联动策略可配置、可解释、可回溯
控制执行对车辆或设备下发控制权限、安全、失败保护
缓存与上报本地缓存、断点续传、云端同步弱网可用、数据完整性
OTA 与配置软件升级、模型升级、参数下发灰度、回滚、版本追踪

5.3 云端模块

模块作用产品关注点
设备管理车辆、网关、传感器、版本管理资产模型、生命周期
数据平台遥测、事件、日志、文件采集数据质量、成本、权限
算法训练数据标注、训练、评估样本闭环、指标口径
OTA 平台软件、固件、模型升级安全、灰度、回滚
运营平台告警、工单、报表、驾驶行为业务闭环和 ROI
策略平台规则、阈值、模型参数配置生效范围和审计

6. 需求拆解方法

6.1 从场景开始，而不是从技术开始

不建议这样写需求：

系统需要支持车载边缘计算能力。

更好的写法：

当车辆在弱网或断网环境下运行时，系统仍需在车端完成关键状态判断，并在网络恢复后补传事件和关键数据。

产品需求要描述：

  • 谁在什么场景下遇到什么问题。
  • 为什么必须在边缘侧处理。
  • 处理结果如何影响用户行为或业务结果。
  • 成功和失败如何定义。

6.2 需求模板

一个车载边缘计算需求可以按以下结构写：

字段说明
场景车辆运行、停车、充电、作业、维保、事故等
角色驾驶员、车队管理员、运维人员、算法团队等
输入数据摄像头、雷达、CAN、GPS、传感器、云端策略
边缘处理过滤、融合、识别、规则判断、缓存、控制
输出结果告警、控制指令、事件、日志、遥测、报表
延迟要求毫秒级、秒级、分钟级
网络要求在线、弱网、离线、断点续传
失败策略重试、降级、人工接管、只读、停止控制
验收指标准确率、延迟、成功率、误报率、完整率

6.3 示例需求：智能灌溉车辆或农机边缘控制

场景：

农机或车载设备在农田弱网环境下，需要根据本地传感器和策略完成灌溉控制。

产品拆解：

  • 输入：土壤湿度、阀门状态、水压、GPS、作业区域、云端策略。
  • 边缘处理：本地规则判断、离线计划执行、异常告警、数据缓存。
  • 输出：阀门控制、执行日志、告警事件、作业记录。
  • 云端协同：策略下发、任务配置、报表分析、远程运维。
  • 失败策略：断网继续执行本地策略，传感器异常则进入安全模式。

验收指标：

  • 离线任务执行成功率。
  • 控制指令响应时延。
  • 数据补传完整率。
  • 误告警率。
  • 现场人工干预次数。

7. 数据闭环

车载边缘计算产品的长期竞争力来自数据闭环。

7.1 数据类型

数据类型示例用途
遥测数据速度、位置、电压、温度、传感器值状态监控和趋势分析
事件数据告警、急刹、碰撞预警、控制执行运营闭环和安全分析
日志数据系统日志、模型日志、通信日志故障定位
文件数据图片、视频片段、雷达数据包算法训练和复盘
配置数据阈值、策略、模型版本策略管理

7.2 数据闭环流程

车端采集
  -> 边缘过滤和压缩
  -> 事件触发上传
  -> 云端存储和分析
  -> 问题定位 / 样本筛选
  -> 算法或策略优化
  -> OTA / 配置下发
  -> 车端效果验证

7.3 产品经理要定义的数据规则

  • 什么数据必须实时上传？
  • 什么数据只在事件触发时上传？
  • 什么数据只保留在车端？
  • 数据保留多久？
  • 谁可以访问？
  • 是否需要用户授权？
  • 是否支持删除、匿名化和最小化留存？
  • 数据如何和车辆、版本、场景、事件关联？

8. 核心指标

8.1 技术指标

指标含义
端到端延迟从数据产生到结果输出的时间
计算耗时单次推理或规则执行耗时
CPU/GPU/NPU 使用率算力资源占用
内存占用运行稳定性和硬件要求
功耗对续航、散热和硬件设计的影响
数据完整率采集和上报是否丢失
OTA 成功率软件、固件、模型升级可靠性

8.2 产品指标

指标含义
功能启用率用户或车辆是否实际使用
告警处理率告警是否被闭环
误报率错误提醒的比例
漏报率应提醒但未提醒的比例
人工干预次数边缘自治是否减少人工介入
故障定位时长运维效率
运营成本下降商业价值

8.3 安全指标

指标含义
降级成功率异常情况下是否进入安全状态
关键链路可用性感知、决策、控制链路是否稳定
异常恢复时间故障后恢复速度
指令审计完整率控制操作是否可追溯
安全事件数失控、误控、危险提示等事件

9. 产品路线图

9.1 MVP 阶段

目标：证明边缘计算能解决明确场景问题。

能力范围：

  • 支持少量核心数据采集。
  • 支持本地规则判断。
  • 支持基础告警。
  • 支持云端配置下发。
  • 支持断网缓存和补传。

不建议一开始就做：

  • 全量数据湖。
  • 复杂模型训练平台。
  • 多车型全适配。
  • 大规模自动驾驶闭环。

9.2 平台化阶段

目标：从单场景复制到多车型、多设备、多项目。

能力范围：

  • 设备模型标准化。
  • 数据字典管理。
  • 策略配置中心。
  • OTA 和版本管理。
  • 远程诊断。
  • 多租户和权限管理。

9.3 智能化阶段

目标：让系统具备持续优化能力。

能力范围：

  • 样本自动筛选。
  • 模型版本管理。
  • A/B 测试。
  • 自动化评估。
  • 预测性维护。
  • 运营策略推荐。

10. 风险和约束

10.1 技术风险

  • 传感器数据质量不稳定。
  • 多传感器时间同步困难。
  • 车端算力和功耗受限。
  • 弱网环境导致数据不完整。
  • OTA 失败导致设备不可用。
  • 不同车型和控制器差异大。

10.2 产品风险

  • 需求只描述技术能力，没有对应业务价值。
  • 告警过多导致用户忽略。
  • 指标定义不清，无法验收。
  • 只重视云端平台，忽略车端异常处理。
  • 数据采集太多，成本高但价值低。
  • 缺少运维工具，现场问题无法定位。

10.3 合规和安全风险

  • 位置数据、驾驶行为、车内图像涉及隐私。
  • 远程控制指令需要严格权限和审计。
  • Modbus、CAN、私有协议等现场协议缺少原生安全机制。
  • 云端策略下发错误可能影响大量车辆。
  • 训练数据和真实用户数据需要分级管理。

11. 竞品和市场分析框架

产品经理做竞品分析时，不建议只比较“谁的算力更强”。更应该比较产品闭环。

维度需要比较的问题
场景覆盖覆盖智能驾驶、车队、座舱、运维还是路侧协同？
车端能力是否支持本地规则、模型推理、缓存、诊断？
云端能力是否有设备管理、OTA、数据分析、策略配置？
开放能力是否提供 API、SDK、数据字典、插件机制？
部署方式前装、后装、私有化、SaaS、边缘网关？
运营闭环是否支持告警、工单、报表和远程运维？
成本结构硬件成本、通信成本、云资源成本、运维成本？

12. 学习路径

12.1 第一阶段：建立概念地图

  • 学习边缘计算、云计算、端侧计算的区别。
  • 理解车端、路侧、云端的分工。
  • 熟悉 CAN、以太网、T-Box、域控制器、OTA 等基本概念。
  • 阅读：边缘计算基本概念、车载边缘计算的特点。

12.2 第二阶段：理解产品场景

  • 拆解一个智能驾驶或车队运营场景。
  • 写出用户、痛点、输入数据、输出结果。
  • 判断哪些逻辑必须在车端完成。
  • 定义延迟、可用性、失败策略和验收指标。

12.3 第三阶段：理解数据和平台

  • 学习设备模型、数据字典、遥测、事件、属性、命令。
  • 理解数据采集、缓存、上报、分析和 OTA 闭环。
  • 学习如何用平台能力支撑多车型、多设备、多项目。
  • 可以结合 Modbus 知识库 理解工业设备接入和点位字典设计。

12.4 第四阶段：形成产品方案能力

  • 能画出端边云协同架构。
  • 能写出一份边缘计算功能 PRD。
  • 能定义指标和验收标准。
  • 能识别安全、隐私、成本和运维风险。
  • 能把技术能力转化为商业价值。

13. 产品经理检查清单

在评审车载边缘计算需求时，可以逐项检查：

  • 是否有明确用户和业务场景？
  • 是否说明为什么必须边缘侧处理？
  • 是否定义输入数据、输出结果和处理逻辑？
  • 是否定义实时性指标？
  • 是否定义弱网、断网、传感器异常时的行为？
  • 是否有数据缓存和补传策略？
  • 是否有远程配置、OTA 和版本回滚？
  • 是否有日志、诊断和问题复盘能力？
  • 是否有权限、审计和安全策略？
  • 是否有可量化验收指标？
  • 是否能支撑后续规模化复制？

14. 一句话总结

车载边缘计算对产品经理的意义，不是多一个技术名词，而是多一种产品设计方法：把实时、安全、成本、体验和数据闭环放在同一张图里，明确哪些事情必须在车端发生，哪些事情适合交给云端，哪些事情需要通过平台长期运营。
