AI 辅助产品经理工作流

从事实采集、攻防共识、原型验证到 PRD 沉淀的产品知识闭环。

产品方法原型PRD知识沉淀

这套工作流的目标不是让 PM 更快地写出一份长 PRD，而是让 PM 更稳定地把「事实、判断、原型、决策、验收」串成一个可复用的产品知识闭环。

方法论概览

过去的需求工作常常从「写 PRD」开始，但 PRD 如果过早出现，很容易把尚未验证的想法包装成确定方案。更稳妥的方式是：先让 AI 帮 PM 收集界面事实、识别系统约束、生成可交互原型，再通过攻防共识收敛方向，最后把已经被验证过的内容沉淀为 PRD。

这套流程可以概括为一句话：

源码和线上界面提供事实，知识库保存上下文，攻防共识暴露风险，原型推动决策，PRD 负责交付协同。

PM 在其中的核心职责也随之变化：不是独自编写完整文档，而是维护问题边界、证据质量、决策过程和验收口径。

01 事实采集 页面 / 接口 / 源码 / 反馈
02 知识沉淀 术语 / 页面 / 数据 / 背景
03 攻防共识 假设 / 风险 / 边界 / 反例
04 原型验证 交互 / 状态 / 异常 / 决策
05 PRD 交付 范围 / 验收 / 测试 / 回写

需求产物 schema

每个需求至少应该能被拆成以下结构，避免 PRD 变成单篇孤立文档。

requirement
  context        # 问题、用户、场景、非目标
  evidence       # 页面、接口、源码、反馈
  prototype      # 关键路径、状态、异常
  attackRegister # 风险、反例、待验证点
  decisionLog    # 取舍、负责人、时间
  prd            # 范围、需求、验收、测试
  wikiBackfill   # 上线后的知识库回写

核心原则

1. 先还原事实，再定义需求

需求不能只来自口头描述。PM 应先通过浏览器、源码、接口和历史文档还原现状，明确当前系统已经支持什么、缺少什么、哪里存在体验或业务断点。

事实来源包括：

  • 前端页面布局、按钮、表单、弹窗、空状态、异常状态
  • 接口请求、响应字段、枚举、错误码、权限状态
  • 项目源码中的路由、组件、状态管理、数据模型
  • 线上行为、用户反馈、历史 issue、已有 PRD 或设计稿

任何进入需求文档的关键判断，都应标记为三类之一：

  • 事实：来自源码、页面、接口、日志或已确认资料
  • 推断：基于事实做出的合理解释
  • 假设：尚未验证，但为了推进讨论暂时成立的前提

2. PRD 不是起点，而是共识结果

PRD 的价值在于帮助设计、研发、测试和业务方协作，而不是替代需求澄清。过早写 PRD 会让团队在错误问题上进行精细化加工。

更合适的顺序是：

  1. 采集事实
  2. 构建上下文
  3. 提出最小可验证方案
  4. 进行攻防审查
  5. 生成交互原型
  6. 让用户或关键干系人决策
  7. 把共识沉淀为 PRD

3. 用原型验证理解，而不是用文档解释想象

当需求还存在歧义时，最小可交互原型比长篇说明更有效。原型不必完整，但必须能验证关键问题，例如：

  • 用户是否能理解入口和任务目标
  • 当前流程是否能覆盖主要场景
  • 字段、状态和异常提示是否符合真实业务
  • 方案是否会引入新的操作负担
  • 是否存在被 PRD 文本掩盖的交互断点

4. 攻防共识优先暴露风险

攻防共识不是为了否定方案，而是为了避免团队过早接受一个看似合理的方向。

推荐使用三角色流程：

Builder 提出方案

定义最小可行方案、核心假设、原型范围和实现路径。

Red Team 攻击方案

审查证据不足、边界遗漏、异常场景、用户价值和实现风险。

Judge 维护共识

维护问题登记表，判断当前方案是否足以进入原型、PRD 或开发。

风险等级建议：

等级含义处理方式
P0阻塞决策或原型验证必须解决
P1高风险，会影响方案有效性必须修正或明确接受
P2可接受风险记录到决策日志
P3轻微问题不阻塞推进

标准操作流程

0任务边界问题、用户、场景、非目标
1事实采集页面、接口、源码、状态
2知识沉淀Wiki、字段、术语、待验证
3攻防共识风险、反例、决策日志
4原型验证路径、状态、异常、反馈
5PRD 沉淀范围、需求、验收、测试
6验收回写实现差异、页面、接口、字段

阶段 0：建立任务边界

目标是回答「这次到底要解决什么问题」。

操作建议：

  • 写清楚用户、场景、触发条件和业务目标
  • 明确本轮不解决的问题
  • 列出已知事实、推断和假设
  • 判断是否需要查看前端页面、接口或源码

建议产出：

  • context.md
  • 问题定义
  • 目标用户与场景
  • 非目标清单

阶段 1：采集界面与接口事实

目标是把当前系统状态从「印象」变成「证据」。

操作建议：

  • 打开目标前端页面，记录页面结构、主要控件和用户路径
  • 观察接口请求、响应字段和错误状态
  • 截取典型状态：默认、加载、成功、失败、空数据、权限不足
  • 将页面元素与接口字段映射起来

建议产出：

  • 页面地图
  • 接口清单
  • 字段与状态枚举
  • 体验断点记录

阶段 2：沉淀到知识库

目标是让需求上下文可以被复用，而不是停留在一次对话里。

操作建议：

  • 将页面布局、接口、数据结构、术语解释沉淀到本地知识库
  • 按项目、模块、页面、接口、业务概念建立索引
  • 对每条重要结论标注来源
  • 对仍未确认的问题建立待验证列表

建议产出：

  • 项目 Wiki
  • 页面说明
  • 接口与字段说明
  • 待验证问题列表

阶段 3：运行攻防共识

目标是让方案在进入 PRD 前先经历一次结构化质询。

推荐流程：

  1. Builder 提出最小方案
  2. Red Team 攻击证据、边界、异常、用户价值和实现风险
  3. Builder 修订方案或承认风险
  4. Judge 判断是否达成共识
  5. 未达成共识时，回到事实采集或原型验证

建议产出：

  • attack-register.md
  • prototype-contract.md
  • decision-log.md

阶段 4：生成并验证可交互原型

目标是用可操作界面验证方案，而不是只靠文字描述。

原型应覆盖：

  • 首屏核心任务
  • 主要用户路径
  • 关键字段与状态
  • 至少一个异常或边界状态
  • 需要用户决策的关键分支

原型不追求完整工程化，但必须能让 PM、设计、研发和业务方讨论同一个对象。

建议产出：

  • prototypes/<feature>/index.html
  • 原型验证记录
  • 用户反馈或干系人反馈

阶段 5：沉淀 PRD

目标是把已经收敛的方案转成可协作交付文档。

PRD 至少包含：

  • 背景与目标
  • 用户与场景
  • 用户故事 / JTBD
  • 需求范围
  • 核心流程
  • 功能需求
  • 非功能需求
  • 风险与依赖
  • 验收标准
  • 测试用例

这里的 PRD 不应重新发明方案，而应引用前面阶段已经形成的事实、原型和决策日志。

阶段 6：进入交付与验收闭环

目标是让需求在实现后回到知识库，形成新的事实基础。

操作建议：

  • 将验收标准拆成可执行检查项
  • 对照页面、接口和源码验证实现是否符合 PRD
  • 记录实现差异：哪些是需求变更，哪些是实现限制
  • 将最终页面、接口和字段变化回写知识库

建议产出：

  • 验收清单
  • 测试场景
  • 实现差异记录
  • 更新后的 Wiki 页面

多 PM 协作场景：GitHub 仓库作为公共知识库

当团队中有多个 PM 同时参与需求工作时，最大的问题不是文档数量不够，而是共识来源不一致。每个 PM 都可能掌握一部分业务背景、用户反馈和系统知识，如果缺少公共知识库，PRD 很容易变成多个局部视角的拼接。

在多 PM 场景下，建议将 GitHub 仓库作为公共知识库，将项目源码作为知识库共识的基础来源。

共识基础：源码优先于记忆

团队应建立一条简单规则：

当文档、记忆和系统行为冲突时，以源码、线上页面、接口和可复现行为作为优先证据。

优先级建议：

  1. 生产环境可复现行为
  2. 当前分支源码、接口定义、数据库结构、配置和测试
  3. 已合并的产品决策记录
  4. 已发布 PRD 和设计稿
  5. 会议纪要、聊天记录和个人经验

这不是否定 PM 经验，而是避免团队围绕不同版本的「我记得」反复争论。

GitHub 中的协作对象

GitHub 对象在 PM 工作流中的作用

Issue需求线索、问题登记、待验证假设

多 PM 分工方式

角色职责
Feature PM负责具体需求闭环，包括事实采集、原型、PRD 和验收
Domain PM维护某个业务域或模块的长期知识库
Review PM从用户价值、范围边界和跨模块影响角度审查
Tech Reviewer确认源码、接口、实现约束和技术风险
QA / Delivery确认验收标准、测试场景和发布影响

分支与 PR 规范

建议每个需求创建独立分支：

pm/<feature-slug>

在分支中维护：

requirements/<feature-slug>/context.md
requirements/<feature-slug>/attack-register.md
requirements/<feature-slug>/prototype-contract.md
requirements/<feature-slug>/decision-log.md
requirements/<feature-slug>/prd.md
prototypes/<feature-slug>/index.html

当需求进入评审时，提交 Pull Request。PR 不只是代码评审，也可以是产品共识评审。

PR 描述建议包含：

  • 本需求解决的问题
  • 已确认事实来源
  • 仍未验证的假设
  • 原型地址或文件路径
  • 关键攻防结论
  • PRD 变更摘要
  • 需要评审者重点确认的问题

知识库写入规则

为了避免多人协作时知识库变成「观点仓库」，建议建立四条写入规则：

  1. 没有来源的结论不能写成事实。
  2. 来自源码、接口、页面的结论要附带引用位置。
  3. 假设可以进入文档，但必须标注验证方式和有效期限。
  4. 已被新实现推翻的旧结论，要保留变更记录，而不是静默覆盖。

多 PM 的攻防共识机制

多 PM 场景下，攻防共识可以从单人模拟升级为多人协作：

阶段参与者目标
Builder 提案Feature PM提出方案、原型范围和成功标准
Red Team 审查Review PM、Domain PM、Tech Reviewer攻击证据、边界、技术约束和跨模块影响
Judge 仲裁需求负责人或产品负责人判断是否可进入 PRD 或开发
PRD 沉淀Feature PM将共识写成可交付文档
验收回写QA / Delivery / Feature PM把实现结果更新回知识库

冲突处理原则

多人协作一定会出现冲突。建议按以下顺序处理：

  • 源码与文档冲突：先确认当前运行版本，再更新文档
  • PRD 与原型冲突：以最新决策日志为准
  • 业务口径冲突：记录到 issue，由产品负责人决策
  • 技术可行性冲突：要求 Tech Reviewer 给出源码或实现依据
  • 验收标准冲突：回到用户场景和成功指标，而不是争论文案

PRD 合并前检查清单

每份 PRD 合并到公共知识库前，建议检查：

  • 是否说明了业务目标和用户场景
  • 是否区分了事实、推断和假设
  • 是否引用了页面、接口、源码或历史决策
  • 是否经过攻防审查
  • 是否存在可交互原型或足够清晰的流程描述
  • 是否定义了范围内和范围外
  • 是否写清楚异常、权限、空状态和边界状态
  • 是否包含验收标准和测试场景
  • 是否标注了风险、依赖和待决问题
  • 是否明确需要回写知识库的实现变化

一句话总结

这套工作流的关键不是让 AI 替 PM 产出更多文档，而是让 PM 借助 AI 和工程事实，把需求从「个人理解」推进到「团队可验证共识」。

个人场景下，它让 PM 更快形成高质量判断；多 PM 场景下，它让 GitHub 仓库成为公共知识库，让项目源码成为共识的底座，让 PRD 成为被验证后的交付协议。
