# 知识库设计原则：让个人知识库更像工作台

> 好的个人知识库不是资料越多越好，而是能在下一次判断、复盘或协作时，快速找回上下文、来源和可执行的下一步。

## 设计读法

PersonalLib 是一个面向产品判断、技术理解和长期复盘的个人知识库。它不是展示型官网，也不是文章归档站。

因此，页面设计和内容组织都服务于三个目标：

- 快速进入主题。
- 保留来源和判断边界。
- 让知识能被继续链接、复用和更新。

## 信息架构原则

### 1. 首页先给路径，不堆清单

首页应回答“我现在该从哪里读起”，而不是把所有文章平铺出来。

当前首页按几类入口组织：

- 推荐阅读路径：帮助读者按目标进入。
- 知识库地图：展示稳定主题。
- 长期关注：说明知识库的持续方向。
- 相关链接：连接 GitHub 与仓库。

### 2. 文章按用途分组

技术笔记、产品方法、知识库维护方法应分组呈现。混在一起会降低读者判断成本，也会让侧边栏越来越像杂物柜。

### 3. 每篇文章只承担一个主问题

一篇文章最好只回答一个核心问题。例如：

| 页面 | 主问题 |
| --- | --- |
| [AI 辅助产品经理工作流](AI_PM_Workflow.md) | PM 如何把事实、原型、决策和 PRD 串成闭环 |
| [车载边缘计算知识](EC_on_Viechle.md) | 车载场景下边缘计算有哪些关键构成 |
| [Modbus 知识库](Modbus.md) | 如何理解和检索 Modbus 协议知识 |

## 页面审美原则

### 1. 标题直接，不写口号

标题应该让读者立刻知道页面回答什么问题。可以清晰，不必花哨。

### 2. 第一屏给出判断

文章开头优先说明：

- 这篇文章的定义或核心观点。
- 它适用于什么场景。
- 读者能拿它做什么。

### 3. 表格只用于比较

表格适合承载映射、对照和检查清单，不适合为了“看起来结构化”而滥用。

### 4. 列表用于扫描，段落用于判断

列表让读者快速定位要点。真正的取舍、背景和判断，应该用短段落说明。

## 反模板味检查

维护知识库时，避免几种常见问题：

- 每篇文章都长成“背景 / 目标 / 价值 / 总结”的固定模板。
- 为了显得完整，补充来源中没有的信息。
- 把不确定判断写成确定事实。
- 用模糊词替代精确条件、指标或边界。
- 首页变成按时间堆叠的文章清单。

## 维护检查清单

新增或更新页面前，可以快速检查：

- 页面是否只回答一个稳定问题。
- 是否能从首页或侧边栏被合理找到。
- 标题是否足够直接。
- 页面是否保留了来源、条件和不确定性。
- 是否有必要加到首页，还是只放侧边栏即可。
- 是否能在未来一次工作中被重新使用。

## 与产品方法的关系

这个知识库的组织方式和 [AI 辅助产品经理工作流](AI_PM_Workflow.md) 是同一套思路：先保存事实，再提炼判断，用结构化方式暴露边界，最后让知识回到下一次工作中。
